# 设计模式

[参考-二十四种设计模式总结](https://blog.csdn.net/qq_46081725/article/details/121553993)

为了降低软件模块的耦合性，提高软件的灵活性、兼容性、可复用性、可维护性与可扩展性，人们从宏观道微观对各种软件系统进行拆分、抽象、组装，确立模块间的交互关系，最终通过归纳、总结，将一些软件模式沉淀下来成为通用的解决方案，这就是设计模式的由来。现在公认的模式有23种，但不能局限于此，在不断实践和沉淀中会有更多优秀巧妙的设计模式产生。

面向对象编程的三大特性：封装、继承、多态。

以语言特性(面向对象的三大特性)为”硬件基础”,再加上软件设计原则的”灵魂”而总结出的一些列设计模式。而这些”灵魂”原则被归纳总结为5种，***单一职责原则(Single Responsibility Principle)，开闭原则(Open/Closed Principle)，里式替换原则(Liskov Substitution Principle)、接口隔离原则(Interface Segregation Principle)和依赖倒置原则(Dependency Inversion Principle)，合起来简称”SOLID”,在加上迪米特法则(law of demeter),总共为六大原则。***

***单一职责***：面向对象以类来划分模块边界，再以方法分割其功能。每个类的职责应该单一，或者方法所完成的功能也应该单一，不是自己的分内之事不该负责。

***开闭原则***：开指的是对扩展开放，闭指的是对修改关闭。要通过抽象去实现，高层抽象的泛化保证了底层实现的多态化扩展，而不需要对现有的系统做反复的修改。对系统的扩展只需要添加新的软件模块，系统模式一旦去顶就不再需改现有的代码。

***里式转换***：在任何父类出现的地方子类也一定可以出现。策略模式就是一个很好的例子。

***接口隔离***：高层接口要独立分化，接口定义时应该尽量拆分成较小的粒度，往往一个接口只对应一个职能。

***依赖倒置***：高层模块只依赖上层抽象，而不直接依赖具体的底层实现，从而达到降低耦合的目的。

***迪米特原则***：一个模块对其他模块应该知之甚少，或者说模块之间彼此英高保持陌生，甚至意识不到对方的存在。以此简单化、最小化模块间的联系，并达到松耦合的目的。门面模式就是极好的例子。

设计的最高境界：
①在设计原则的指导下，我们可以对设计模式进行适度地改造、组合，来应为各种复杂的业务场景。但是注意不要滥用设计模式，以免陷入”为了设计而设计”的误区，导致设计过度。
②世界上并不存在无所不能的设计，而任何事物都有其两面性，任何一种设计模式都有其优缺点，所以对设计模式的运用一定要适可而止，否则系统臃肿不堪，在满足当前需求的前提下，并在未来可预估业务范围内的设计才是最合理的设计。
③在系统不能满足需求时我们还可以做出适当的重构，这样的设计才切合实际。

在工作中我们一定要合理地利用设计模式去解决目前以及可以预见的未来所面临的问题，并基于设计原则，不断反复思考与总结。直到有一天，我们可能忘记者写设计模式的名字，突破”招式”和”套路”的牵绊，最终达到一种融会贯通的状态，各种组合拳信手拈来，运用自如。当各种模式在我们的设计种变得”你中有我，我中有你”时，才达到不拘泥于形式的境界。
